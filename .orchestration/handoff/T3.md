## Bye Semantics
- A bye is a one-sided matchup row: home team populated, away team `NULL`, away score `0`, winner `unknown`.
- Stats emits one weekly row for the bye team with `result='bye'`, `opponent_person_id NULL`, PF equal to the provider score, and PA `0`.
- Season PF and scoring records include the bye score.
- Wins, losses, ties, win percentage, streaks, H2H meetings, all-play W/L/T, and AI game-final recap planning do not count the bye as a game.
- The model is ready for a future Data-layer "count byes as wins" option, but default behavior is no W/L/T.

## Span Semantics
- `persistNormalizedLeagueRows` upserts `league_season_settings` before matchups so span derivation can read the stored season settings.
- For matchups inside the configured playoff window, `league_season_settings.playoff_matchup_period_length` is authoritative when present.
- This both raises short provider spans to the setting and clamps over-broad ESPN windows to the setting.
- Regular-season/non-playoff rows keep the provider-normalized span unless a steward edit has made the matchup span sticky.
- Single-week high/low record candidates filter to `scoring_period_span=1`; multi-week totals remain in weekly/season facts and divide averages by span.

## Gotchas For T4+
- Any query/render path reading `fantasy_matchups.away_team_provider_id` must handle `NULL`.
- Any query/render path reading `weekly_statistics.result` must handle `bye`.
- Data-editor work should keep bye W/L/T policy as a league/data-layer setting; do not hard-code count-as-win behavior into the stats engine.
- If T4+ introduces pushed snapshots, snapshot rows must preserve nullable-away matchups and the stored `scoring_period_span`.
- Commissioner edits to `scoring_period_span` are sticky through `league_data_edits`; that remains the override path for future Data Book editing.
- The real import harness uses `.env.local` ESPN cookies. Do not print cookie values in logs or summaries.
