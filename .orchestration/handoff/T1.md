# T1 Handoff - Per-Season Settings Persistence

## What T1 Built
- Extended the existing `league_season_settings` table instead of creating a parallel table.
- ESPN `mSettings` now normalize and persist for both current sync and historical `leagueHistory` imports.

## Settings Schema
`league_season_settings` is keyed by `(league_id, provider, league_provider_id, season)` and remains RLS-scoped.

T1-added fields:
- `league_size`
- `playoff_matchup_period_length`
- `scoring_type`
- `lineup_slot_counts`
- `acquisition_type`
- `acquisition_budget`
- `acquisition_settings`

Existing fields still used:
- `matchup_period_count`
- `regular_season_end_scoring_period`
- `playoff_start_scoring_period`
- `championship_scoring_period`
- `playoff_team_count`
- `scoring_settings`
- keeper flags/settings

## Key Decisions
- Acquisition stores both the explicit type flag and budget. Budget alone is not treated as proof of FAAB.
- `matchup_period_count` remains the ESPN regular-season length signal; `regular_season_end_scoring_period` mirrors it where ESPN supplies it.
- `playoff_matchup_period_length` is stored separately so T3 can derive multi-week playoff spans without re-reading ESPN.
- Current sync and history use the same `upsertLeagueSeasonSettings` path and content hash, so re-imports are idempotent.
- Sticky commissioner/data-steward edits are preserved for new settings fields just like older settings fields.
- ESPN live current schedule rows can omit `schedule[].id`; T1 derives a stable fallback id from matchup period and team ids.
- ESPN live history contains one-sided bye rows. T1 skips those rows during matchup normalization so settings import can complete; T3 still owns real bye representation and coverage semantics.

- Do not remove the one-sided schedule row skip unless T3 is ready to persist byes as first-class one-sided facts.
