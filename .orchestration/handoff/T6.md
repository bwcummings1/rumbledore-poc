# T6 Handoff - Editable Data Book Cells + Scope Prompt

## What T6 Built
- Extended the existing `/leagues/[leagueId]/data` Data Book view; no parallel view or extra destination was added.
- Added steward-level editable dimension cells:
  - People grid: `person.canonical_name` real name.
  - People grid: `team_season.team_name` for the selected season row.
  - Weeks grid: team name where it appears in the team cell.
- Added the required scope prompt after inline confirmation:
  - real name default: `all_years`.
  - team name default: `this_year_only`.
  - both are overridable.
- Submits edits to `POST /api/leagues/[leagueId]/curation/edits` with the selected `scope`; includes `season` only for
  `this_year_only`.
- Successful edits update the local Data Book draft state immediately and mark affected editable cells with a small
  `Draft` pill plus a compact draft notice. They do not save checkpoints, push snapshots, or affect Records.
- Non-stewards still load the Data Book, but receive no edit buttons or inline edit forms.

## Permission Model
- The page still requires only `minRole: "member"` so members can read the Data Book.
- `DataBookView` receives `canEditData` from the server page.
- Current UI edit access is granted to `commissioner`, `league_admin`, and `data_steward`, matching the existing
  guard ranking where `/curation/edits` requires `minRole: "data_steward"`.
- The API remains the enforcement boundary.

## UI Flow
1. Steward clicks a cell edit icon.
2. The cell becomes an inline input with confirm/cancel icon buttons.
3. Confirm opens the shared `Dialog` scope prompt.
4. Steward keeps or changes the `Select` scope, optionally enters a reason, then applies the draft edit.
5. The local Data Book state updates from the returned `affectedTargetIds` and `scope`.

Shared primitives used: `Button`, `Dialog`, `Field`, `Input`, `Select`, `StatusPill`, and existing `DataTable` /
`DataCardTable` responsive behavior.

## API Payload Shape
Real-name all-years default:
```json
{
  "editClass": "cosmetic",
  "field": "canonical_name",
  "reason": "Corrected real name from Data Book",
  "scope": "all_years",
  "targetId": "<person.id>",
  "targetKind": "person",
  "value": "Corrected Name"
}
```

Team-name this-year default:
```json
{
  "editClass": "cosmetic",
  "field": "team_name",
  "reason": "Corrected team name from Data Book",
  "scope": "this_year_only",
  "season": 2026,
  "targetId": "<team_season.id>",
  "targetKind": "team_season",
  "value": "Corrected Team"
}
```

## Tests Added
- Real-name scope prompt defaults to all-years, posts the expected payload, omits `season`, renders the edited value,
  and shows Draft state.
- Team-name scope prompt defaults to this-year-only, posts `scope` plus selected `season`, and renders the edit.
- Team-name override to all-years posts without `season` and updates all affected team seasons locally.
- Non-steward Data Book renders read-only with no edit affordances.
- Route test verifies explicit this-year `scope`/`season` reaches `applyCuratedDataEdit`.
- DB-backed curated-state test verifies ledger `beforeValue` / `afterValue` / `scope` for a scoped Data Book-style edit.

## Gotchas For T7/T8
- T7 should build the ledger/change-log as a separate destination. Do not add the feed to the Data Book.
- T8 can hook into the same Data Book route for save/push controls, but T6 does not create checkpoints or pushes.
- Data Book local state is optimistic only after the edit API succeeds. A fresh navigation reloads from draft tables.
- `DataTable` renders desktop and mobile markup in the DOM during tests; duplicate edit buttons/text are expected.
- For `person.canonical_name` + `this_year_only`, T4 may split the person and return a new person id in
  `affectedTargetIds`; T6 updates local rows using both returned person ids and team-season ids.
