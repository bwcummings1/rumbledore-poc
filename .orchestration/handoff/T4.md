# T4 Handoff - Curated-State Model + Scoped Edits

## What T4 Built
- Added the draft -> saved checkpoint -> pushed snapshot data/service layer.
- Added scoped dimension edits:
  - real name (`person.canonical_name`) smart-defaults to `all_years`.
  - team name (`team_season.team_name`) smart-defaults to `this_year_only`.
  - both defaults are overridable with `scope: "all_years" | "this_year_only"`.
- Added checkpoint APIs for save/list/restore.
- Added push APIs for per-season push and pushAll.
- Added `composeCanonicalSnapshot(leagueId)`, the future T9 record-book input.
- Did **not** re-point the existing record book. Current record pages still read live/materialized stats until T9.

## Schema
- `league_data_edits.scope text NULL`
  - Valid values: `all_years`, `this_year_only`, or `NULL` for older/non-scoped audit rows.
  - New ledger target kinds:
    - `curation_checkpoint`
    - `curation_push`
- `league_curation_checkpoints`
  - RLS-scoped, `FORCE ROW LEVEL SECURITY`, append-only trigger.
  - One row per save. Keeps all checkpoints.
  - Stores a whole-league draft `snapshot` JSON, `seasons`, `edit_ids`, `latest_edit_id`, `snapshot_hash`.
  - Anchored to `league_data_edits` by `marker_edit_id` (`field='checkpoint_save'`).
- `league_curation_season_pushes`
  - RLS-scoped, `FORCE ROW LEVEL SECURITY`, append-only trigger.
  - One row per season push. Keeps all pushed versions.
  - Stores one season `snapshot` JSON, `season`, `checkpoint_id`, `edit_ids`, `latest_edit_id`, `snapshot_hash`.
  - Anchored to `league_data_edits` by `marker_edit_id` (`field='season_push'`).

## Service API
Exports live from `src/stats/index.ts`.

- `applyCuratedDataEdit(db, input)`
  - Scope-aware wrapper around the existing curation edit behavior.
  - Existing `applyLeagueDataEdit` still exists for old callers.
  - `person.canonical_name` + `this_year_only` splits/remaps that season to a new `person` row so other seasons remain on the original person.
  - `team_season.team_name` + `all_years` propagates the team name to all team seasons mapped to the same person.
- `createCurationCheckpoint(db, { leagueId, actorUserId, label?, note? })`
  - Save. Captures the current draft state. Not canonical.
- `listCurationCheckpoints(db, { leagueId, limit? })`
  - Lists checkpoint summaries newest-first.
- `restoreCurationCheckpoint(db, { leagueId, actorUserId, checkpointId, reason? })`
  - Restores draft tables, including season groupings, from the checkpoint and writes a restore marker.
  - Does not delete checkpoint rows.
- `pushCurationSeason(db, { leagueId, actorUserId, season, checkpointId?, reason? })`
  - Pushes one season from the specified checkpoint, or the latest saved checkpoint if omitted.
- `pushAllCurationSeasons(db, { leagueId, actorUserId, checkpointId?, reason? })`
  - Pushes every season in the specified/latest checkpoint.
- `composeCanonicalSnapshot(db, { leagueId })`
  - Returns only pushed data: the latest pushed row for each season, flattened plus per-season snapshots.

## Composition Invariant
The canonical pushed state is the composition of every season's latest-pushed version.

Pushing one season must never replace the whole canonical snapshot. It inserts a new row for that season only.
`composeCanonicalSnapshot` then selects the latest push per season, so pushing 2012 leaves 2011 and every other
previously pushed season intact. Saved checkpoints are invisible until a push row exists for their season.

This is covered by `src/stats/curated-state.test.ts`.

## API Routes
- `POST /api/leagues/[leagueId]/curation/edits`
  - Now calls `applyCuratedDataEdit`.
  - Optional request fields: `scope`, `season`.
- `GET /api/leagues/[leagueId]/curation/checkpoints`
  - Lists saved checkpoints.
- `POST /api/leagues/[leagueId]/curation/checkpoints`
  - Creates a save checkpoint.
- `POST /api/leagues/[leagueId]/curation/checkpoints/[checkpointId]/restore`
  - Restores a checkpoint.
- `POST /api/leagues/[leagueId]/curation/push`
  - `{ action: "push", season, checkpointId?, reason? }`
  - `{ action: "pushAll", checkpointId?, reason? }`

All checkpoint/push mutations require `minRole: "data_steward"`.

## Gotchas
- Checkpoints are whole-league snapshots; pushes are per-season snapshots.
- `pushCurationSeason` promotes the latest saved checkpoint when `checkpointId` is omitted. It does not push unsaved draft edits.
- `restoreCurationCheckpoint` restores draft tables only. It does not rewrite history or delete checkpoint/push rows.
- Existing live record-book materializations may still reflect live edits because T9 has not re-pointed records to `composeCanonicalSnapshot`.
- Snapshot JSON preserves nullable-away bye matchups and `scoring_period_span`.
- Do not use `Promise.all` inside `withLeagueContext` transactions when extending this service.

## What T5/T8/T9 Need To Know
- T5 Data Book read UI should read live draft tables for the editable Data layer, not the composed canonical snapshot.
- T8 save/push UI should call:
  - save: `POST /curation/checkpoints`
  - restore: `POST /curation/checkpoints/:checkpointId/restore`
  - push season: `POST /curation/push` with `action: "push"`
  - push all: `POST /curation/push` with `action: "pushAll"`
- T8 should present saved checkpoint state as draft-only and pushed state as canonical.
- T9 should re-point record-book reads to `composeCanonicalSnapshot(leagueId)` and preserve the invariant that every season's latest pushed contribution remains accounted for.
